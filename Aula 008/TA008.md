## 1. Conceitos Fundamentais

### O Problema da Escala e Alta Disponibilidade

Imagine que sua aplicação Docker está rodando perfeitamente em um único servidor, mas agora você precisa:
- **Escalar** para atender mais usuários
- **Alta disponibilidade** - se um servidor falhar, a aplicação continua rodando
- **Load balancing** automático entre múltiplas instâncias
- **Rolling updates** sem downtime

Gerenciar isso manualmente com `docker run` em múltiplos servidores seria um pesadelo. É aí que entra o **Docker Swarm**.

### O que é Docker Swarm?

**Docker Swarm** é a solução nativa do Docker para **orquestração de containers** em cluster. Ele transforma um grupo de máquinas Docker em um único "super-computador" virtual que pode gerenciar containers automaticamente.

**Características principais:**
- **Nativo do Docker**: Já vem integrado, sem instalação adicional
- **Simples**: Mais fácil que Kubernetes para começar
- **Declarativo**: Você define o estado desejado, Swarm mantém
- **Auto-healing**: Substitui containers que falham automaticamente
- **Load balancing**: Distribui tráfego automaticamente

### Arquitetura do Swarm

#### Manager Nodes (Nós Gerenciadores)
Os **Manager Nodes** são o "cérebro" do cluster:
- Gerenciam o estado do cluster
- Tomam decisões de scheduling
- Expõem a API do Swarm
- Mantêm consenso via algoritmo Raft

```bash
# Inicializar swarm (torna-se manager)
docker swarm init

# Adicionar manager adicional
docker swarm join-token manager
```

#### Worker Nodes (Nós Trabalhadores)
Os **Worker Nodes** executam os containers:
- Recebem tarefas dos managers
- Executam containers
- Reportam status de volta aos managers

```bash
# Obter token para workers
docker swarm join-token worker

# Worker se junta ao cluster
docker swarm join --token SWMTKN-1-xxx... 192.168.1.100:2377
```

#### Recomendações de Arquitetura
- **Managers**: Sempre número ímpar (1, 3, 5, 7)
- **Quorum**: Maioria dos managers deve estar online
- **Produção**: Mínimo 3 managers para alta disponibilidade

### Services vs Tasks vs Containers

#### Service (Serviço)
Um **Service** é a definição de como você quer que sua aplicação rode:
```bash
# Criar serviço com 3 réplicas
docker service create --name web --replicas 3 -p 80:80 nginx
```

#### Task (Tarefa)
Uma **Task** é uma instância específica de um serviço em um nó:
- Cada réplica = uma task
- Task é imutável (se falha, nova task é criada)

#### Container
O **Container** é a execução real da task no nó.

```
Service: web (3 réplicas)
├── Task 1 → Container no Node A
├── Task 2 → Container no Node B
└── Task 3 → Container no Node C
```

### Comandos Essenciais do Swarm

#### Gerenciamento do Cluster
```bash
# Inicializar swarm
docker swarm init --advertise-addr 192.168.1.100

# Listar nós
docker node ls

# Inspecionar nó
docker node inspect node-name

# Promover worker a manager
docker node promote worker-node

# Remover nó do cluster
docker node rm node-name
```

#### Gerenciamento de Services
```bash
# Criar serviço
docker service create --name web --replicas 3 -p 80:80 nginx

# Listar serviços
docker service ls

# Inspecionar serviço
docker service inspect web

# Ver tarefas do serviço
docker service ps web

# Escalar serviço
docker service scale web=5

# Atualizar serviço
docker service update --image nginx:alpine web

# Remover serviço
docker service rm web
```

### Redes no Swarm

#### Overlay Networks
**Overlay networks** conectam containers em diferentes nós:

```bash
# Criar rede overlay
docker network create --driver overlay --attachable minha-rede

# Serviço usando rede overlay
docker service create \
  --name web \
  --network minha-rede \
  --replicas 3 \
  nginx
```

#### Ingress Network
A rede **ingress** fornece load balancing automático:
- Qualquer nó pode receber tráfego na porta publicada
- Tráfego é automaticamente roteado para container saudável
- Load balancing round-robin por padrão

### Docker Stack: Compose para Swarm

#### Arquivo docker-stack.yml
```yaml
version: '3.8'

services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"
    deploy:
      replicas: 3
      update_config:
        parallelism: 1
        delay: 10s
      restart_policy:
        condition: on-failure
    networks:
      - webnet

  api:
    image: minha-api:latest
    deploy:
      replicas: 2
      placement:
        constraints:
          - node.role == worker
    networks:
      - webnet
      - backend

  database:
    image: postgres:13
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
    secrets:
      - db_password
    deploy:
      replicas: 1
      placement:
        constraints:
          - node.labels.database == true
    networks:
      - backend
    volumes:
      - db-data:/var/lib/postgresql/data

networks:
  webnet:
    driver: overlay
  backend:
    driver: overlay

volumes:
  db-data:

secrets:
  db_password:
    external: true
```

#### Comandos de Stack
```bash
# Deploy da stack
docker stack deploy -c docker-stack.yml minha-app

# Listar stacks
docker stack ls

# Listar serviços da stack
docker stack services minha-app

# Ver tarefas da stack
docker stack ps minha-app

# Remover stack
docker stack rm minha-app
```

### Secrets e Configs

#### Secrets (Dados Sensíveis)
```bash
# Criar secret
echo "senha-super-secreta" | docker secret create db_password -

# Listar secrets
docker secret ls

# Usar secret no serviço
docker service create \
  --name db \
  --secret db_password \
  postgres:13
```

#### Configs (Configurações)
```bash
# Criar config
docker config create nginx_config nginx.conf

# Usar config no serviço
docker service create \
  --name web \
  --config source=nginx_config,target=/etc/nginx/nginx.conf \
  nginx
```

### Rolling Updates e Rollbacks

#### Rolling Update
```bash
# Atualizar imagem gradualmente
docker service update \
  --image nginx:1.21 \
  --update-parallelism 2 \
  --update-delay 30s \
  web
```

#### Rollback
```bash
# Voltar para versão anterior
docker service rollback web

# Rollback automático em caso de falha
docker service update \
  --image nginx:1.21 \
  --update-failure-action rollback \
  --rollback-parallelism 1 \
  web
```

### Placement Constraints

#### Constraints por Papel
```yaml
deploy:
  placement:
    constraints:
      - node.role == manager  # Apenas em managers
      - node.role == worker   # Apenas em workers
```

#### Constraints por Labels
```bash
# Adicionar label ao nó
docker node update --label-add database=true node-1

# Usar constraint no serviço
deploy:
  placement:
    constraints:
      - node.labels.database == true
```

## 2. Questões de Múltipla Escolha

### Questão 1
Qual é a principal diferença entre Docker Compose e Docker Swarm?

a) Compose é para desenvolvimento, Swarm é para produção em cluster
b) Compose é mais rápido que Swarm
c) Swarm só funciona no Linux
d) Não há diferença, são a mesma ferramenta

**Gabarito: A**

**Justificativa:** Docker Compose é ideal para desenvolvimento e ambientes single-host, definindo aplicações multi-container em uma máquina. Docker Swarm é para produção em cluster, orquestrando containers em múltiplas máquinas com alta disponibilidade, load balancing automático e healing. Swarm usa arquivos similares ao Compose, mas com recursos adicionais de orquestração.

### Questão 2
Por que é recomendado ter um número ímpar de Manager Nodes em um cluster Swarm?

a) Para economizar recursos de hardware
b) Para garantir quorum e evitar split-brain em caso de falha de rede
c) Porque números pares causam conflitos
d) Para melhor performance de load balancing

**Gabarito: B**

**Justificativa:** O algoritmo de consenso Raft usado pelo Swarm requer maioria (quorum) dos managers para tomar decisões. Com número ímpar, sempre há uma maioria clara em caso de split-brain (partição de rede). Com 3 managers, se 1 falha, ainda há maioria (2/3). Com número par, uma partição 50/50 impediria decisões, causando indisponibilidade do cluster.

### Questão 3
No Docker Swarm, o que acontece quando você publica uma porta de um serviço com múltiplas réplicas?

a) Apenas a primeira réplica recebe tráfego
b) Você precisa configurar load balancer externo manualmente
c) O Swarm automaticamente distribui tráfego entre todas as réplicas saudáveis
d) Cada réplica precisa de uma porta diferente

**Gabarito: C**

**Justificativa:** O Docker Swarm possui load balancing automático através da rede ingress. Quando você publica uma porta (`-p 80:80`), qualquer nó do cluster pode receber tráfego nessa porta e o Swarm automaticamente roteia para uma réplica saudável usando round-robin. Isso elimina a necessidade de configurar load balancers externos para distribuição básica de tráfego.
