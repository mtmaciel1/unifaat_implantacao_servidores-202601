# Lab 9: Introdução à Orquestração (Docker Swarm)

**Disciplina:** Implementação de servidor e nuvem (cloud)

**Objetivo:** Inicializar um Cluster Swarm, criar um Service escalável, provar a distribuição de carga (Load Balancing) e o gerenciamento do ciclo de vida das réplicas.

- Demonstrar tolerância a falhas e autorrecuperação

- Monitorar e inspecionar nós e réplicas


### Pré-requisitos

* Docker Desktop rodando e integrado ao WSL/Linux.
* Conhecimento dos comandos básicos do Docker.

## Cenário: Deploy de uma Aplicação Web Escalável (Cluster de Nó Único)
Vamos simular o ambiente Swarm usando nosso Host como o único nó (Manager).

## Seção 1: Inicialização do Cluster Swarm

1.  **Verificar Status:** Verifique se o Swarm já está ativo (se estiver, use `docker swarm leave --force`).
```bash
docker info | grep Swarm
```
2.  **Inicializar o Swarm:** Inicialize o *cluster* no Host atual. Este Host se tornará o nó **Manager**.
```bash
docker swarm init
```
3.  **Verificar Nós (Nodes):** Confirme se o nó (seu Host) está ativo e no papel `Manager`.
```bash
docker node ls
```

## Seção 2: Criação e Escalabilidade do Service

O *Service* é a definição da nossa aplicação rodando no *cluster*.

1.  **Criar o Service:** Crie um *Service* chamado **`web-scaling-9`** usando a imagem `nginx:alpine`.
    * Publique a porta **8005** do *cluster* para a porta 80 do Nginx (`-p 8005:80`).
    * Defina o Service para começar com 3 réplicas (`--replicas 3`).
```bash
docker service create \
--name web-scaling-9 \
--replicas 3 \
-p 8005:80 \
nginx:alpine
```
    
2.  **Monitorar o Deploy:** Verifique se as 3 réplicas foram lançadas com sucesso.
```bash
docker service ps web-scaling-9
```

3.  **Acesso e Load Balancing:** Acesse a porta `8005` algumas vezes. O Swarm garante que, mesmo em um nó único, a conexão será roteada para uma das 3 réplicas ativas.
```bash
curl localhost:8005
```

## Seção 3: Prova de Escalabilidade

Vamos aumentar a capacidade do nosso *Service* em tempo real.

1.  **Aumentar Escala:** Redefina o número de réplicas de 3 para **5**.
```bash
docker service scale web-scaling-9=5
```
2.  **Monitorar o Aumento:** Observe o *status* do *Service*. O Swarm deve ter lançado mais duas réplicas.
```bash
docker service ps web-scaling-9
```

## Seção 4: Prova de Tolerância a Falhas (Simulação)

Mesmo em um nó único, o Swarm tenta manter a contagem de réplicas desejada.

1.  **Identificar uma Réplica:** Use o comando `docker service ps` para obter o **ID do Contêiner** de uma das réplicas (por exemplo, a réplica mais antiga).
2.  **Remover o Contêiner Diretamente (Falha Simuleda):** Force a remoção de um dos contêineres usando o `docker rm`.
```bash
docker rm -f [ID_DO_CONTÊINER]
```
3.  **Observar a Autorrecuperação:** Imediatamente, execute o comando `docker service ps web-scaling-9` novamente.

*(**Resultado Esperado:** O Swarm detecta que o número de réplicas caiu para 4 (ou menos) e **automaticamente lança uma nova réplica** para restaurar a contagem desejada (5).)*

## Seção 5: Limpeza Final do Cluster

É crucial remover o *Service* e sair do *Cluster* Swarm.

1.  **Remover o Service:** O Swarm irá remover todas as 5 réplicas associadas.
```bash
docker service rm web-scaling-9
```
2.  **Sair do Swarm:** Saia do modo Swarm (`docker swarm leave`). Use a *flag* `--force` se o Host for o Manager e houverem recursos pendentes.
```bash
docker swarm leave --force
```
3.  **Verificação Final:** Confirme que o Swarm está inativo.
```bash
docker info | grep Swarm
```

## Dicas Importantes

1. **Swarm é ideal** para ambientes de produção com Docker nativo
2. **Escale facilmente** com `docker service scale`

4. **Load balancing** é automático entre réplicas
5. **Sempre limpe** removendo services antes de sair do cluster
