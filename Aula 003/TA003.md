## 1. Conceitos Fundamentais

### O que é um Dockerfile e Por que Precisamos Dele?

Imagine que você desenvolveu uma aplicação incrível no seu computador. Ela funciona perfeitamente! Mas agora você precisa que ela rode no computador do seu colega, no servidor de produção, e na máquina do cliente. O problema é que cada ambiente pode ter:
- Versões diferentes de Python, Node.js ou Java
- Bibliotecas diferentes instaladas
- Sistemas operacionais diferentes (Windows, Linux, macOS)
- Configurações diferentes

O **Dockerfile** resolve esse problema sendo uma "receita universal" que garante que sua aplicação rode **exatamente igual** em qualquer lugar.

### Analogia: Dockerfile como Receita de Bolo

Pense no Dockerfile como uma receita de bolo muito detalhada:

**Receita Tradicional:**
- "Misture os ingredientes"
- "Asse no forno"

**Receita Dockerfile (Detalhada):**
- "Use uma forma de 20cm (imagem base)"
- "Adicione 200g de farinha (instalar dependências)"
- "Misture por 5 minutos (executar comandos)"
- "Asse a 180°C por 30 minutos (configurar aplicação)"

Com uma receita detalhada, qualquer pessoa pode fazer o mesmo bolo. Com um Dockerfile, qualquer computador pode rodar sua aplicação.

### Anatomia de um Dockerfile: Passo a Passo

Vamos construir um Dockerfile do zero para uma aplicação web Python simples:

#### 1. FROM - Escolhendo a Base

```dockerfile
FROM python:3.9-slim
```

**O que faz:** Define a "base" da sua imagem. É como escolher o tipo de massa para sua pizza.

**Analogia:** Se você vai fazer uma pizza, precisa decidir: massa tradicional, integral, ou sem glúten? No Docker, você escolhe: Ubuntu, Alpine Linux, ou uma imagem já com Python instalado?

**Opções comuns:**
- `python:3.9-slim` - Python já instalado, imagem pequena
- `ubuntu:20.04` - Ubuntu limpo, você instala tudo
- `node:16-alpine` - Node.js em Alpine Linux (muito pequeno)
- `nginx:alpine` - Servidor web Nginx pronto

#### 2. WORKDIR - Definindo o "Local de Trabalho"

```dockerfile
WORKDIR /app
```

**O que faz:** Define onde você vai "trabalhar" dentro do container. É como escolher em qual cômodo da casa você vai cozinhar.

**Analogia:** Você não cozinha na sala ou no quarto, você vai para a cozinha. No container, você não coloca arquivos em qualquer lugar, você define um diretório específico.

**Por que é importante:**
- Organização: todos os arquivos ficam em um lugar
- Segurança: não mistura com arquivos do sistema
- Facilita manutenção: você sempre sabe onde estão seus arquivos

#### 3. COPY - Trazendo Seus Arquivos

```dockerfile
# Primeiro, copie apenas o arquivo de dependências
COPY requirements.txt .

# Depois de instalar dependências, copie o resto
COPY . .
```

**O que faz:** Copia arquivos do seu computador para dentro da imagem Docker.

**Analogia:** É como levar os ingredientes da despensa para a bancada da cozinha antes de cozinhar.

**Estratégia Inteligente:**
```dockerfile
# ❌ Ruim - copia tudo de uma vez
COPY . .
RUN pip install -r requirements.txt

# ✅ Bom - copia dependências primeiro
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
```

**Por que essa ordem é melhor?**
Se você mudar apenas o código (não as dependências), o Docker reutiliza a instalação das bibliotecas do cache, tornando o build muito mais rápido!

#### 4. RUN - Executando Comandos

```dockerfile
# Atualizar sistema e instalar dependências
RUN apt-get update && apt-get install -y \
    gcc \
    && rm -rf /var/lib/apt/lists/*

# Instalar bibliotecas Python
RUN pip install --no-cache-dir -r requirements.txt
```

**O que faz:** Executa comandos durante a construção da imagem. É como seguir os passos da receita.

**Analogia:** "Pré-aqueça o forno a 180°C", "Misture os ingredientes secos", "Bata as claras em neve".

**Boas Práticas:**
```dockerfile
# ❌ Ruim - cada RUN cria uma camada
RUN apt-get update
RUN apt-get install -y python3
RUN apt-get install -y pip
RUN apt-get clean

# ✅ Bom - um RUN com múltiplos comandos
RUN apt-get update && \
    apt-get install -y python3 pip && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

#### 5. EXPOSE - Documentando Portas

```dockerfile
EXPOSE 5000
```

**O que faz:** Documenta qual porta sua aplicação usa. É como colocar uma placa na porta: "Aplicação Web - Porta 5000".

**Importante:** EXPOSE não publica a porta automaticamente! É apenas documentação. Para publicar, você usa `-p` no `docker run`.

#### 6. CMD - O Comando Final

```dockerfile
CMD ["python", "app.py"]
```

**O que faz:** Define o comando que será executado quando o container iniciar.

**Analogia:** É como o último passo da receita: "Sirva quente com sorvete".

### Exemplo Completo: Aplicação Flask

Vamos criar um Dockerfile completo para uma aplicação web Python:

**Estrutura do projeto:**
```
meu-projeto/
├── app.py
├── requirements.txt
├── Dockerfile
└── templates/
    └── index.html
```

**app.py:**
```python
from flask import Flask, render_template

app = Flask(__name__)

@app.route('/')
def home():
    return render_template('index.html')

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000, debug=True)
```

**requirements.txt:**
```
Flask==2.3.3
```

**Dockerfile:**
```dockerfile
# 1. Escolher a base - Python já instalado
FROM python:3.9-slim

# 2. Informações sobre quem criou (opcional, mas boa prática)
LABEL maintainer="seu-email@exemplo.com"
LABEL description="Minha primeira aplicação Flask"

# 3. Definir diretório de trabalho
WORKDIR /app

# 4. Copiar arquivo de dependências primeiro (para cache)
COPY requirements.txt .

# 5. Instalar dependências Python
RUN pip install --no-cache-dir -r requirements.txt

# 6. Copiar todo o código da aplicação
COPY . .

# 7. Criar usuário não-root (segurança)
RUN adduser --disabled-password --gecos '' appuser && \
    chown -R appuser:appuser /app
USER appuser

# 8. Documentar que a aplicação usa porta 5000
EXPOSE 5000

# 9. Comando para iniciar a aplicação
CMD ["python", "app.py"]
```

### Construindo e Testando a Imagem

```bash
# 1. Construir a imagem
docker build -t minha-flask-app .

# 2. Verificar se a imagem foi criada
docker images

# 3. Executar um container
docker run -d -p 5000:5000 --name meu-container minha-flask-app

# 4. Testar se funciona
curl http://localhost:5000

# 5. Ver logs do container
docker logs meu-container

# 6. Parar e remover
docker stop meu-container
docker rm meu-container
```

### .dockerignore - O que NÃO Incluir

Assim como o `.gitignore` evita que arquivos desnecessários vão para o Git, o `.dockerignore` evita que arquivos desnecessários vão para a imagem Docker.

**.dockerignore:**
```
# Arquivos de desenvolvimento
.git/
.gitignore
README.md
Dockerfile
.dockerignore

# Dependências locais
node_modules/
__pycache__/
*.pyc

# Arquivos de IDE
.vscode/
.idea/
*.swp

# Logs e arquivos temporários
*.log
.tmp/
.cache/

# Arquivos sensíveis
.env
secrets.txt
*.key
```

**Por que usar .dockerignore?**
- **Segurança:** Evita copiar arquivos sensíveis
- **Performance:** Imagens menores = builds mais rápidos
- **Limpeza:** Apenas o necessário vai para produção

### Multi-Stage Builds: Imagens Inteligentes

Imagine que você precisa compilar uma aplicação Go. Você precisa do compilador Go para fazer o build, mas não precisa dele para rodar a aplicação final.

**Problema:** Se você incluir o compilador na imagem final, ela fica muito grande.
**Solução:** Multi-stage builds!

```dockerfile
# Stage 1: Build (imagem temporária com ferramentas de desenvolvimento)
FROM golang:1.19 AS builder

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download

COPY . .
RUN go build -o main .

# Stage 2: Production (imagem final pequena)
FROM alpine:latest

# Instalar certificados SSL (necessário para HTTPS)
RUN apk --no-cache add ca-certificates
WORKDIR /root/

# Copiar APENAS o executável do stage anterior
COPY --from=builder /app/main .

# Executar a aplicação
CMD ["./main"]
```

**Resultado:** 
- Imagem de build: ~800MB (com Go compiler)
- Imagem final: ~10MB (apenas o executável)

### Otimização de Imagens: Dicas Práticas

#### 1. Escolha Imagens Base Pequenas

```dockerfile
# ❌ Imagem grande (1GB+)
FROM ubuntu:20.04

# ✅ Imagem pequena (~5MB)
FROM alpine:latest

# ✅ Meio termo - funcionalidade + tamanho
FROM python:3.9-slim  # ~100MB vs python:3.9 (~900MB)
```

#### 2. Combine Comandos RUN

```dockerfile
# ❌ Cada RUN cria uma camada (imagem maior)
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y wget
RUN apt-get clean

# ✅ Um RUN com tudo (imagem menor)
RUN apt-get update && \
    apt-get install -y curl wget && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

#### 3. Use Cache de Dependências

```dockerfile
# ✅ Ordem otimizada para cache
COPY package.json package-lock.json ./  # Arquivos de dependência primeiro
RUN npm ci --only=production            # Instalar dependências
COPY . .                                # Código por último
```

#### 4. Remova Arquivos Desnecessários

```dockerfile
RUN apt-get update && \
    apt-get install -y build-essential && \
    # ... compilar aplicação ... && \
    apt-get remove -y build-essential && \  # Remover ferramentas de build
    apt-get autoremove -y && \              # Remover dependências órfãs
    apt-get clean && \                      # Limpar cache do apt
    rm -rf /var/lib/apt/lists/*             # Remover listas de pacotes
```

### Segurança no Dockerfile

#### 1. Não Use Root

```dockerfile
# Criar usuário não-privilegiado
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nextjs -u 1001

# Mudar para usuário não-root
USER nextjs
```

#### 2. Não Inclua Secrets

```dockerfile
# ❌ NUNCA faça isso
ENV API_KEY=abc123secret

# ✅ Use variáveis de ambiente no runtime
ENV API_KEY=""
```

#### 3. Use Imagens Oficiais

```dockerfile
# ✅ Imagens oficiais são mais seguras
FROM node:16-alpine
FROM python:3.9-slim
FROM nginx:alpine

# ❌ Evite imagens não oficiais
FROM random-user/custom-python
```

### Debugging de Dockerfiles

#### Construir Passo a Passo

```bash
# Ver cada camada sendo construída
docker build --progress=plain -t minha-app .

# Parar em uma camada específica para debug
docker build --target builder -t debug-image .
docker run -it debug-image /bin/bash
```

#### Inspecionar Imagem

```bash
# Ver histórico de camadas
docker history minha-app

# Ver detalhes da imagem
docker inspect minha-app

# Ver tamanho de cada camada
docker images --format "table {{.Repository}}\t{{.Tag}}\t{{.Size}}"
```

### Exemplo Avançado: Aplicação Node.js com Nginx

**Estrutura do projeto:**
```
projeto-completo/
├── frontend/
│   ├── package.json
│   ├── src/
│   └── public/
├── backend/
│   ├── package.json
│   ├── server.js
│   └── routes/
├── nginx/
│   └── nginx.conf
└── docker-compose.yml
```

**Dockerfile para Frontend (React):**
```dockerfile
# Multi-stage build para React
FROM node:16-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

COPY . .
RUN npm run build

# Nginx para servir arquivos estáticos
FROM nginx:alpine
COPY --from=builder /app/build /usr/share/nginx/html
COPY nginx/nginx.conf /etc/nginx/nginx.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**Dockerfile para Backend (Node.js API):**
```dockerfile
FROM node:16-alpine

# Criar usuário não-root
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodeuser -u 1001

WORKDIR /app

# Instalar dependências como root
COPY package*.json ./
RUN npm ci --only=production && \
    npm cache clean --force

# Copiar código e mudar ownership
COPY . .
RUN chown -R nodeuser:nodejs /app

# Mudar para usuário não-root
USER nodeuser

EXPOSE 3000

# Health check
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

CMD ["node", "server.js"]
```

## 2. Questões de Múltipla Escolha

### Questão 1
Por que é recomendado copiar o arquivo `package.json` (ou `requirements.txt`) antes de copiar todo o código da aplicação no Dockerfile?

a) Por questões de segurança, para não expor o código-fonte
b) Para aproveitar o sistema de cache de camadas do Docker e acelerar builds subsequentes
c) Porque o arquivo de dependências deve estar na raiz da imagem
d) Para evitar conflitos entre diferentes versões de bibliotecas

**Gabarito: B**

**Justificativa:** O Docker constrói imagens em camadas e usa cache para acelerar builds. Ao copiar `package.json` primeiro e instalar dependências, essa camada fica em cache. Quando você modifica apenas o código (não as dependências), o Docker reutiliza a camada de instalação do cache, tornando builds subsequentes muito mais rápidos. Se copiasse tudo junto, qualquer mudança no código invalidaria o cache das dependências, forçando reinstalação desnecessária.

### Questão 2
Qual é a principal vantagem de usar multi-stage builds em Dockerfiles?

a) Permite usar múltiplos sistemas operacionais na mesma imagem
b) Reduz significativamente o tamanho final da imagem separando ferramentas de build dos artefatos de produção
c) Aumenta a velocidade de execução dos containers
d) Permite executar múltiplas aplicações simultaneamente no mesmo container

**Gabarito: B**

**Justificativa:** Multi-stage builds permitem usar uma imagem "pesada" com todas as ferramentas de desenvolvimento (compiladores, build tools, dependências de desenvolvimento) para construir a aplicação, e depois copiar apenas os artefatos necessários (executáveis, arquivos compilados) para uma imagem "leve" de produção. Isso resulta em imagens finais drasticamente menores - por exemplo, uma aplicação Go pode ir de 800MB (com compilador) para 10MB (apenas executável).

### Questão 3
Qual é a diferença prática entre as instruções `CMD` e `ENTRYPOINT` no Dockerfile?

a) `CMD` é para sistemas Linux e `ENTRYPOINT` é para Windows
b) `CMD` pode ser facilmente sobrescrito no `docker run`, enquanto `ENTRYPOINT` define um comando mais fixo
c) `CMD` executa durante a construção da imagem, `ENTRYPOINT` executa durante a execução do container
d) Não há diferença prática, são sinônimos e podem ser usados intercambiavelmente

**Gabarito: B**

**Justificativa:** `CMD` define o comando padrão que pode ser facilmente sobrescrito passando argumentos no `docker run`. `ENTRYPOINT` define um comando mais "fixo" - argumentos passados no `docker run` são adicionados ao `ENTRYPOINT`, não o substituem completamente. Por exemplo: se `ENTRYPOINT ["python"]` e `CMD ["app.py"]`, executar `docker run minha-imagem debug.py` resultará em `python debug.py`. É comum usar `ENTRYPOINT` + `CMD` juntos para ter flexibilidade controlada.
