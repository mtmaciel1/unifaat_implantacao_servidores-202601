# Lab 5: Redes no Docker

**Disciplina:** Implementação de servidor e nuvem (cloud)  
**Aula:** 5 - Redes no Docker  
**Objetivo:** Criar uma Rede Customizada (`bridge`), provar o **DNS Interno** (comunicação por nome) e utilizar comandos de inspeção de rede.

## Objetivo do Lab

Ao final deste laboratório, você será capaz de:
- Compreender as limitações da rede bridge padrão
- Criar e gerenciar redes customizadas no Docker
- Implementar DNS interno para comunicação entre contêineres
- Testar conectividade de rede entre serviços
- Inspecionar configurações de rede para troubleshooting
- Aplicar isolamento de rede entre diferentes serviços

### Pré-requisitos

- Docker Desktop rodando e integrado ao WSL/Linux.
- Conhecimento dos comandos básicos do Docker (`run`, `ps`, `exec`).

## Seção 1: O Problema da Rede Padrão (Verificação)
Vamos provar que contêineres na rede `bridge` padrão não se comunicam por nome.

### Passo 1.1: Lançamento na Rede Padrão

1.  Rode um contêiner Nginx com um nome simples (rede `bridge` padrão).
 ```bash
 docker run -d --name servico-web-A nginx:latest
 ```

2.  Rode um contêiner Alpine (imagem leve com ferramentas de rede) para teste.
 ```bash
 docker run -d --name servico-tester-B alpine:latest sleep infinity        
```

3.  Verifique se ambos estão ativos:
```bash
docker ps
```

### Passo 1.2: Teste de Conectividade (FALHA ESPERADA)
Tente usar o `ping` do `servico-tester-B` para acessar o `servico-web-A` pelo **nome**.

1.  Entre no terminal do `servico-tester-B` e execute o `ping`:
```bash
docker exec -it servico-tester-B ping servico-web-A
```
*(**Resultado Esperado:** O comando deve falhar com "ping: bad address 'servico-web-A'").*

2.  **Limpeza:** Pare e remova os contêineres:
```bash
docker stop servico-web-A servico-tester-B
docker rm servico-web-A servico-tester-B
```

## Seção 2: Solução com Rede Customizada (DNS Interno)
A Rede Customizada (Driver `bridge`) permite que os contêineres se encontrem usando o nome do serviço (DNS Interno).

### Passo 2.1: Criação da Rede Customizada

1.  Crie uma rede customizada chamada `rede-lab5`:
```bash
docker network create rede-lab5
```
2.  Liste as redes para confirmar que ela foi criada:
```bash
docker network ls
```

### Passo 2.2: Lançamento de Contêineres na Rede

Lance novamente os dois serviços, mas **anexando-os** à rede customizada.

1.  Rode o contêiner Nginx (Web) anexado à rede:
```bash
docker run -d --name servico-web-customizado --network rede-lab5 nginx:latest
 ```
2.  Rode o contêiner Alpine (Tester) anexado à mesma rede:
```bash
docker run -d --name servico-tester-customizado --network rede-lab5 alpine:latest sleep infinity
```

### Passo 2.3: Teste de Conectividade (SUCESSO ESPERADO)

Tente novamente usar o `ping` do `servico-tester-customizado` para acessar o `servico-web-customizado` pelo **nome**.

1.  Entre no terminal do `servico-tester-customizado` e execute o `ping`:
```bash
docker exec -it servico-tester-customizado ping servico-web-customizado
```
*(**Resultado Esperado:** O comando deve ter sucesso, exibindo os pacotes de resposta, provando que o DNS Interno está funcionando!)*

2.  Pare o `ping` com `Ctrl + C`.

## Seção 3: Inspeção e Detalhes de Rede
Para fins de *troubleshooting* (resolução de problemas), é vital saber como inspecionar a rede.

### Passo 3.1: Inspeção da Rede e Endereços IP

1.  Use o comando de inspeção para visualizar todos os detalhes da `rede-lab5`. Procure a seção `"Containers"` para ver qual IP foi atribuído a cada um.
```bash
docker network inspect rede-lab5
```
2.  **Pergunta:** Qual é o *Driver* de rede principal utilizado (listado na saída do `inspect`)?

### Passo 3.2: Limpeza Final

1.  Pare e remova os contêineres:
```bash
docker stop servico-web-customizado servico-tester-customizado
```
2.  Remova a rede customizada (só é possível se não houver contêineres anexados):
```bash
docker network rm rede-lab5
```

## Dicas Importantes

1. **Use redes customizadas** para comunicação por nome entre contêineres
2. **Rede bridge padrão** não oferece DNS interno automático
3. **Sempre inspecione redes** com `docker network inspect` para debug
4. **Isole serviços** usando redes separadas para segurança
5. **Limpe redes não utilizadas** com `docker network prune`
