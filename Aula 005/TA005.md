## 1. Conceitos Fundamentais

### O Grande Problema: Containers Isolados Como Ilhas

Imagine que você tem uma cidade onde cada casa (container) está em uma **ilha separada**. Como as pessoas de uma casa visitam a outra? Como o padeiro entrega pão para o açougueiro? Como o banco se comunica com a loja?

Por padrão, containers Docker são como essas ilhas - **completamente isolados** uns dos outros. Isso é ótimo para segurança (ninguém invade a casa do vizinho), mas cria um problema: **como fazer containers se comunicarem quando necessário?**

### Cenário Real: E-commerce Quebrado

Vamos imaginar um e-commerce simples:

```
🌐 Frontend (React) ←→ 🔧 Backend (API) ←→ 🗄️ Database (MySQL)
```

Se cada um está em um container isolado:
- Frontend não consegue chamar a API
- API não consegue acessar o banco de dados
- **Resultado**: Site completamente quebrado! 💥

**É aí que entram as Redes Docker!**

### Analogia: Redes Como Sistema de Transporte

Pense nas redes Docker como diferentes **sistemas de transporte** entre as ilhas:

- **Bridge (Ponte)**: Uma ponte conectando algumas ilhas específicas
- **Host (Teleporte)**: Containers "teleportam" para o computador host
- **None (Isolamento Total)**: Ilha completamente isolada, sem transporte
- **Custom (Ponte Privada)**: Sua própria ponte personalizada com regras específicas

### Os Quatro Tipos de Redes Docker

#### 1. Bridge Network (A Ponte Padrão) 🌉

**Bridge** é como uma ponte automática que o Docker cria para conectar containers:

```bash
# Docker cria automaticamente uma rede bridge padrão
docker run -d --name web nginx
docker run -d --name api node:16

# Os containers podem se comunicar, mas apenas por IP
docker exec web ping 172.17.0.3  # IP do container 'api'
```

**Características da Bridge Padrão:**
- ✅ **Automática**: Docker cria sozinho
- ✅ **Isolamento**: Separado da rede do host
- ❌ **Sem DNS**: Containers não se encontram por nome
- ❌ **Limitada**: Todos os containers na mesma "ponte"

**Analogia**: É como uma ponte pública onde todos podem passar, mas ninguém tem endereço - você precisa decorar onde cada pessoa mora.

#### 2. Host Network (O Teleporte) 🚀

**Host** remove completamente o isolamento - o container usa diretamente a rede do seu computador:

```bash
# Container "teleporta" para a rede do host
docker run -d --network host nginx

# Nginx agora está diretamente no localhost:80
curl http://localhost:80
```

**Características da Host:**
- ✅ **Performance Máxima**: Sem overhead de rede
- ✅ **Simplicidade**: Acesso direto ao host
- ❌ **Sem Isolamento**: Menos seguro
- ❌ **Conflitos de Porta**: Containers podem conflitar

**Analogia**: É como convidar alguém para morar na sua casa - super conveniente, mas você perde privacidade.

**Quando usar Host:**
- Aplicações que precisam de performance máxima de rede
- Ferramentas de monitoramento que precisam acessar a rede do host
- Desenvolvimento rápido (mas não recomendado para produção)

#### 3. None Network (Isolamento Total) 🏝️

**None** desabilita completamente a rede do container:

```bash
# Container completamente isolado
docker run -d --network none alpine

# Container não tem acesso à internet nem a outros containers
docker exec container_id ping google.com  # Falha!
```

**Características da None:**
- ✅ **Máxima Segurança**: Zero acesso à rede
- ✅ **Processamento Puro**: Ideal para cálculos
- ❌ **Sem Comunicação**: Não acessa nada
- ❌ **Limitado**: Apenas processamento local

**Analogia**: É como uma sala cofre - super segura, mas você não consegue falar com ninguém.

**Quando usar None:**
- Processamento de dados sensíveis
- Cálculos matemáticos que não precisam de rede
- Containers que apenas processam arquivos locais

#### 4. Custom Bridge (A Ponte Inteligente) 🧠

**Custom Bridge** é sua própria rede personalizada com superpoderes:

```bash
# Criar sua própria rede
docker network create minha-rede

# Containers na rede personalizada
docker run -d --network minha-rede --name web nginx
docker run -d --network minha-rede --name api node:16

# Agora eles se encontram por nome! 🎉
docker exec web ping api  # Funciona!
docker exec api curl http://web:80  # Funciona!
```

**Características da Custom Bridge:**
- ✅ **DNS Automático**: Containers se encontram por nome
- ✅ **Isolamento**: Apenas containers autorizados
- ✅ **Flexibilidade**: Configurações personalizadas
- ✅ **Segurança**: Controle total sobre quem se comunica

**Analogia**: É como criar um condomínio privado com portaria - apenas moradores autorizados entram, e todos se conhecem pelo nome.

### Comandos Essenciais de Redes

#### Gerenciamento Básico
```bash
# 1. Listar todas as redes
docker network ls

# 2. Criar rede personalizada
docker network create minha-app-network

# 3. Ver detalhes de uma rede
docker network inspect minha-app-network

# 4. Conectar container existente à rede
docker network connect minha-app-network meu-container

# 5. Desconectar container da rede
docker network disconnect minha-app-network meu-container

# 6. Remover rede (cuidado!)
docker network rm minha-app-network

# 7. Limpeza - remover redes não utilizadas
docker network prune
```

#### Configurações Avançadas
```bash
# Rede com subnet personalizada (como definir endereços IP)
docker network create \
  --subnet=192.168.100.0/24 \
  --ip-range=192.168.100.128/25 \
  --gateway=192.168.100.1 \
  rede-personalizada

# Container com IP fixo
docker run -d \
  --network rede-personalizada \
  --ip 192.168.100.10 \
  --name web-server \
  nginx

# Rede com driver específico
docker network create \
  --driver bridge \
  --opt com.docker.network.bridge.name=minha-bridge \
  rede-especial
```

### Exemplo Prático Completo: E-commerce Funcional

Vamos criar um e-commerce completo que funciona de verdade:

#### Estrutura da Aplicação
```
E-commerce
├── 🌐 Frontend (React) - Porta 3000
├── 🔧 Backend (Node.js API) - Porta 5000  
├── 🗄️ Database (MySQL) - Porta 3306
└── 📦 Cache (Redis) - Porta 6379
```

#### Passo 1: Criar a Rede
```bash
# Criar rede para o e-commerce
docker network create ecommerce-network

# Verificar se foi criada
docker network ls | grep ecommerce
```

#### Passo 2: Banco de Dados MySQL
```bash
# Container MySQL
docker run -d \
  --name ecommerce-db \
  --network ecommerce-network \
  -e MYSQL_ROOT_PASSWORD=senha123 \
  -e MYSQL_DATABASE=ecommerce \
  -e MYSQL_USER=app \
  -e MYSQL_PASSWORD=appsenha \
  -v mysql-data:/var/lib/mysql \
  mysql:8.0

# Verificar se está rodando
docker ps | grep ecommerce-db
```

#### Passo 3: Cache Redis
```bash
# Container Redis
docker run -d \
  --name ecommerce-cache \
  --network ecommerce-network \
  redis:alpine

# Testar conexão
docker exec ecommerce-cache redis-cli ping
```

#### Passo 4: Backend API
```bash
# Container da API (Node.js)
docker run -d \
  --name ecommerce-api \
  --network ecommerce-network \
  -p 5000:5000 \
  -e DB_HOST=ecommerce-db \
  -e DB_USER=app \
  -e DB_PASSWORD=appsenha \
  -e DB_NAME=ecommerce \
  -e REDIS_HOST=ecommerce-cache \
  minha-api:latest

# Testar API
curl http://localhost:5000/health
```

#### Passo 5: Frontend React
```bash
# Container do Frontend
docker run -d \
  --name ecommerce-frontend \
  --network ecommerce-network \
  -p 3000:3000 \
  -e REACT_APP_API_URL=http://ecommerce-api:5000 \
  meu-frontend:latest

# Acessar aplicação
# http://localhost:3000
```

#### Passo 6: Testar Comunicação
```bash
# Frontend consegue acessar API?
docker exec ecommerce-frontend curl http://ecommerce-api:5000/health

# API consegue acessar banco?
docker exec ecommerce-api ping ecommerce-db

# API consegue acessar cache?
docker exec ecommerce-api ping ecommerce-cache

# Todos devem responder! 🎉
```

### Resolução DNS: A Mágica dos Nomes

#### Como Funciona o DNS em Redes Personalizadas

Quando você cria uma rede personalizada, o Docker automaticamente configura um **servidor DNS interno**:

```bash
# Criar rede
docker network create app-network

# Containers na rede
docker run -d --network app-network --name web nginx
docker run -d --network app-network --name api node:16

# Docker automaticamente configura:
# web → 172.18.0.2
# api → 172.18.0.3

# Containers podem usar nomes em vez de IPs
docker exec web curl http://api:3000
docker exec api curl http://web:80
```

#### Exemplo Prático: Aplicação WordPress

```bash
# 1. Criar rede para WordPress
docker network create wordpress-net

# 2. Banco de dados
docker run -d \
  --name wp-database \
  --network wordpress-net \
  -e MYSQL_ROOT_PASSWORD=rootpass \
  -e MYSQL_DATABASE=wordpress \
  -e MYSQL_USER=wpuser \
  -e MYSQL_PASSWORD=wppass \
  mysql:5.7

# 3. WordPress (usa nome do container para conectar ao DB)
docker run -d \
  --name wp-site \
  --network wordpress-net \
  -p 8080:80 \
  -e WORDPRESS_DB_HOST=wp-database:3306 \
  -e WORDPRESS_DB_USER=wpuser \
  -e WORDPRESS_DB_PASSWORD=wppass \
  -e WORDPRESS_DB_NAME=wordpress \
  wordpress:latest

# WordPress automaticamente encontra o banco pelo nome "wp-database"!
```

### Mapeamento de Portas: Conectando com o Mundo Exterior

#### Sintaxes de Mapeamento
```bash
# Sintaxe básica: -p HOST_PORT:CONTAINER_PORT
docker run -p 8080:80 nginx  # localhost:8080 → container:80

# Múltiplas portas
docker run -p 8080:80 -p 8443:443 nginx

# Bind em interface específica
docker run -p 127.0.0.1:8080:80 nginx  # Apenas localhost

# Porta aleatória do host
docker run -P nginx  # Docker escolhe porta automaticamente

# Ver qual porta foi escolhida
docker port container_name
```

#### Exemplo: Aplicação Multi-Porta
```bash
# Aplicação com múltiplos serviços
docker run -d \
  --name app-completa \
  --network minha-rede \
  -p 3000:3000 \  # Frontend
  -p 5000:5000 \  # API
  -p 8080:8080 \  # Admin
  -p 9090:9090 \  # Métricas
  minha-app-completa
```

### Segurança em Redes Docker

#### Isolamento por Ambiente
```bash
# Rede de produção
docker network create prod-network

# Rede de desenvolvimento  
docker network create dev-network

# Rede de testes
docker network create test-network

# Containers isolados por ambiente
docker run -d --network prod-network --name prod-api minha-api
docker run -d --network dev-network --name dev-api minha-api
docker run -d --network test-network --name test-api minha-api

# Cada ambiente é completamente isolado!
```

#### Controle de Acesso
```bash
# Rede apenas para serviços internos
docker network create --internal internal-services

# Containers nesta rede não acessam a internet
docker run -d --network internal-services --name internal-api minha-api

# Testar (deve falhar)
docker exec internal-api ping google.com  # Sem acesso à internet
```

#### Firewall com iptables
```bash
# Ver regras que Docker criou
sudo iptables -L DOCKER

# Docker automaticamente configura:
# - NAT para containers
# - Isolamento entre redes
# - Regras de port mapping
```

### Microserviços: Arquitetura Real

#### Cenário: Sistema de Pedidos
```bash
# 1. Rede para microserviços
docker network create microservices-net

# 2. Serviço de Usuários
docker run -d \
  --name user-service \
  --network microservices-net \
  -e DB_HOST=user-db \
  user-api:latest

# 3. Banco do serviço de usuários
docker run -d \
  --name user-db \
  --network microservices-net \
  -e POSTGRES_DB=users \
  postgres:13

# 4. Serviço de Pedidos
docker run -d \
  --name order-service \
  --network microservices-net \
  -e DB_HOST=order-db \
  -e USER_SERVICE_URL=http://user-service:3000 \
  order-api:latest

# 5. Banco do serviço de pedidos
docker run -d \
  --name order-db \
  --network microservices-net \
  -e POSTGRES_DB=orders \
  postgres:13

# 6. API Gateway (ponto de entrada)
docker run -d \
  --name api-gateway \
  --network microservices-net \
  -p 80:80 \
  -e USER_SERVICE=http://user-service:3000 \
  -e ORDER_SERVICE=http://order-service:3000 \
  gateway:latest
```

#### Fluxo de Comunicação
```
Internet → API Gateway → User Service → User DB
            ↓
         Order Service → Order DB
```

### Debugging de Redes

#### Comandos de Diagnóstico
```bash
# 1. Inspecionar rede detalhadamente
docker network inspect minha-rede

# 2. Ver quais containers estão na rede
docker network inspect minha-rede | grep Name

# 3. Testar conectividade entre containers
docker exec container1 ping container2

# 4. Ver configuração de rede de um container
docker exec container1 ip addr show

# 5. Testar resolução DNS
docker exec container1 nslookup container2

# 6. Ver rotas de rede
docker exec container1 ip route show
```

#### Troubleshooting Comum

**Problema 1: Containers não se comunicam**
```bash
# Verificar se estão na mesma rede
docker inspect container1 | grep NetworkMode
docker inspect container2 | grep NetworkMode

# Solução: colocar na mesma rede
docker network connect minha-rede container1
docker network connect minha-rede container2
```

**Problema 2: DNS não funciona**
```bash
# Verificar se é rede personalizada (bridge padrão não tem DNS)
docker network ls

# Solução: criar rede personalizada
docker network create minha-rede-dns
docker run --network minha-rede-dns --name container1 nginx
```

**Problema 3: Porta não acessível**
```bash
# Verificar mapeamento de portas
docker port meu-container

# Verificar se container está rodando
docker ps | grep meu-container

# Verificar logs
docker logs meu-container
```

### Exemplo Avançado: Aplicação Completa com Load Balancer

```bash
# 1. Rede para aplicação
docker network create webapp-network

# 2. Banco de dados
docker run -d \
  --name webapp-db \
  --network webapp-network \
  -e POSTGRES_DB=webapp \
  -e POSTGRES_USER=app \
  -e POSTGRES_PASSWORD=secret \
  postgres:13

# 3. Múltiplas instâncias da API (para load balancing)
docker run -d --name api-1 --network webapp-network -e DB_HOST=webapp-db api:latest
docker run -d --name api-2 --network webapp-network -e DB_HOST=webapp-db api:latest
docker run -d --name api-3 --network webapp-network -e DB_HOST=webapp-db api:latest

# 4. Load Balancer (Nginx)
docker run -d \
  --name load-balancer \
  --network webapp-network \
  -p 80:80 \
  -v $(pwd)/nginx.conf:/etc/nginx/nginx.conf \
  nginx

# 5. Frontend
docker run -d \
  --name frontend \
  --network webapp-network \
  -p 3000:3000 \
  -e API_URL=http://load-balancer \
  frontend:latest
```

**nginx.conf para Load Balancing:**
```nginx
upstream api_backend {
    server api-1:3000;
    server api-2:3000;
    server api-3:3000;
}

server {
    listen 80;
    
    location /api/ {
        proxy_pass http://api_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

## 2. Questões de Múltipla Escolha

### Questão 1
Você tem uma aplicação web (frontend) que precisa se comunicar com uma API (backend), ambos em containers separados. Qual é a melhor abordagem para configurar a comunicação entre eles?

a) Usar a rede bridge padrão e comunicar por endereços IP
b) Criar uma rede personalizada e conectar ambos containers a ela
c) Usar a rede host para ambos os containers
d) Usar bind mounts para compartilhar arquivos entre containers

**Gabarito: B**

**Justificativa:** Criar uma rede personalizada é a melhor prática porque oferece **resolução DNS automática** (containers se encontram por nome), **isolamento** (apenas containers autorizados podem se comunicar), e **flexibilidade** de configuração. A rede bridge padrão não oferece DNS por nome, a rede host remove isolamento de segurança, e bind mounts são para compartilhar arquivos, não para comunicação de rede.

### Questão 2
Qual é a principal vantagem de usar redes Docker personalizadas em vez da rede bridge padrão?

a) Redes personalizadas são mais rápidas que a rede bridge padrão
b) Redes personalizadas oferecem resolução DNS automática por nome do container
c) Redes personalizadas consomem menos recursos do sistema
d) Redes personalizadas só funcionam em ambientes de produção

**Gabarito: B**

**Justificativa:** A principal vantagem das redes personalizadas é a **resolução DNS automática**, permitindo que containers se comuniquem usando nomes em vez de endereços IP. Na rede bridge padrão, containers precisam usar IPs (que podem mudar) ou links legados (deprecated). Com DNS automático, você pode usar `http://meu-backend:3000` em vez de `http://172.17.0.3:3000`, tornando a comunicação mais robusta e fácil de manter.

### Questão 3
Em um ambiente de microserviços, você quer garantir que apenas containers autorizados possam se comunicar entre si. Qual estratégia de rede é mais adequada?

a) Colocar todos os containers na rede bridge padrão
b) Usar rede host para todos os containers
c) Criar redes personalizadas separadas para cada grupo de serviços relacionados
d) Usar rede none para máximo isolamento

**Gabarito: C**

**Justificativa:** Criar **redes personalizadas separadas** para grupos de serviços relacionados oferece o melhor equilíbrio entre comunicação e segurança. Por exemplo: rede "frontend" para UI e API Gateway, rede "backend" para APIs e bancos de dados, rede "monitoring" para ferramentas de monitoramento. Isso garante que apenas serviços que precisam se comunicar tenham acesso uns aos outros, implementando o princípio de menor privilégio em nível de rede.
