## Seção 1: Bind Mounts (Foco em Desenvolvimento Local)

O **Bind Mount** é a ferramenta ideal para desenvolvimento, pois permite que o código editado no Host seja refletido instantaneamente no Contêiner.

### Passo 1.1: Preparação do Ambiente e Código Simulado

No seu terminal **WSL/Linux**:

1.  Crie um diretório de trabalho chamado `lab_volume_app`.
```bash
mkdir ~/aulas_lab/aula004
cd ~/aulas_lab/aula004
```

2. Abra o VSCode na pasra atual
```bash
code .
```

3.  Crie um arquivo HTML simples (index.html) que será servido pelo Nginx.
```html
<h1>
    Versao 1.0 - Inicial
</h1>
```

3.  Verifique o caminho absoluto (necessário para o Bind Mount).
```bash
pwd
 ```

### Passo 1.2: Lançamento do Contêiner com Bind Mount

Vamos lançar um contêiner Nginx, montando o diretório atual do seu Host para o diretório padrão de arquivos estáticos do Nginx (`/usr/share/nginx/html`).

1.  Lance o contêiner em *detached mode* (`-d`), mapeando a porta do Host (ex: 8080) e usando o Bind Mount (`-v`).
```bash
docker run -d --name dev_nginx -p 8080:80 -v $(pwd):/usr/share/nginx/html nginx:latest
```
> **Este comando Docker executa um contêiner Nginx com as seguintes configurações:**<br>
> **-d :** Executa o contêiner em modo "detached" (em background)
>
> **--name dev_nginx :** Define o nome do contêiner como "dev_nginx"
>
> **-p 8080:80 :** Mapeia a porta 8080 do host para a porta 80 do contêiner. Permite acessar o Nginx através de localhost:8080
>
> **-v \$(pwd) /usr/share/nginx/html :** Cria um bind mount  - Este mapeamento permite editar arquivos no host e ver mudanças refletidas no contêiner
>
> **\$(pwd) :** Diretório atual do host
> **/usr/share/nginx/html :** Diretório dentro do contêiner onde o Nginx serve arquivos estáticos
>
> **nginx:latest :** Usa a imagem mais recente do Nginx do Docker Hub

2.  Verifique se o contêiner está rodando.
    ```bash
    docker ps   
    ```
3.  Abra seu navegador e acesse: `http://localhost:8080`.

### Passo 1.3: Prova de Persistência em Tempo Real (Bind Mount)

1.  **Edite** o arquivo `index.html`:
```html
<h1>
    Versao 2.0 - Codigo Atualizado!
</h1>
```
2.  **Verifique no navegador** (`http://localhost:8080`) se a mudança foi instantânea, provando que o contêiner está lendo diretamente do disco do Host.

## Seção 2: Volumes Nomeados (Foco em Produção e DBs)

**Volumes Nomeados** são ideais para bancos de dados, pois o Docker gerencia o ciclo de vida e o armazenamento, garantindo que os dados persistam mesmo após a remoção do contêiner.

### Passo 2.1: Criação e Inspeção do Volume Nomeado

1.  Crie um Volume Nomeado para armazenar dados do PostgreSQL:
```bash
docker volume create db_volume_lab4
```

2.  Liste os volumes para confirmar a criação:
```bash
docker volume ls
```

3.  **Comando de Inspeção Chave:** Use o `docker volume inspect` para ver onde o Docker está armazenando o volume no Host (Procure pelo campo `Mountpoint`).
```bash
docker volume inspect db_volume_lab4
```

### Passo 2.2: Lançamento do Contêiner com Volume
Lance um contêiner PostgreSQL, garantindo que ele use o volume criado para persistir seus dados.

1.  Lance o contêiner do Postgres (`-d`) mapeando o volume para o diretório de dados padrão do Postgres (`/var/lib/postgresql/data`).
```bash
docker run -d --name lab4_db -e POSTGRES_PASSWORD=mysecretpassword -v db_volume_lab4:/var/lib/postgresql/data postgres:16
```
2.  Verifique se o contêiner está ativo.
```bash
docker ps
```
### Passo 2.3: criar dados do Banco de dados
1.  Acesse o terminal do contêiner PostgreSQL:
```bash
docker exec -it lab4_db psql -U postgres
```

2. Conectado ao terminal do container PostgreSQL como usuário postgres

*A partir daqui podemos executar comandos SQL di*retamente no banco de dados*

```sql
-- Verificar se os dados ainda estão presentes
SELECT * FROM Alunos;

-- Criar tabela Alunos
CREATE TABLE Alunos (
    RA varchar(10) PRIMARY KEY,
    Nome varchar(100)
);

-- Inserir 4 alunos
INSERT INTO Alunos (RA, Nome) VALUES ('1234567', 'João Silva');
INSERT INTO Alunos (RA, Nome) VALUES ('2345678', 'Maria Santos');
INSERT INTO Alunos (RA, Nome) VALUES ('3456789', 'Pedro Oliveira');
INSERT INTO Alunos (RA, Nome) VALUES ('4567890', 'Ana Pereira');

-- Consultar dados inseridos
SELECT * FROM Alunos;
```


### Passo 2.3: Prova de Persistência (Ciclo de Vida Independente)

Para provar que o volume é independente do contêiner, vamos forçar a remoção do contêiner e relançá-lo.

1.  **Parar e Remover o Contêiner:**
```bash
docker stop lab4_db
docker rm lab4_db
```
*Neste ponto, a Camada de Escrita do contêiner foi destruída, mas o volume `db_volume_lab4` ainda existe.*

2.  **Lançar um Novo Contêiner (Usando o Mesmo Volume):** Lance um novo contêiner (com um nome diferente, mas usando o **MESMO VOLUME**).
 ```bash
docker run -d --name lab4_db_novo -e POSTGRES_PASSWORD=mysecretpassword -v db_volume_lab4:/var/lib/postgresql/data postgres:16
```
3.  **Análise:** vamos verificar se os dados continuam lá
```bash
docker exec -it lab4_db_novo psql -U postgres
```
4. Agora que estamos conectado no postgres vamos verificar os dados
```sql
-- Verificar se os dados ainda estão presentes
SELECT * FROM Alunos;
```
5. Vamos sair ca conexão
```bash
\q
```

Ao finalizar o Lab, execute os seguintes comandos no seu terminal WSL para limpar todos os recursos:


## Dicas Importantes

1. **Use bind mounts para desenvolvimento** - permite hot reload do código
2. **Use volumes nomeados para produção** - Docker gerencia o ciclo de vida
3. **Sempre inspecione volumes** com `docker volume inspect` para debug
4. **Volumes são independentes** do ciclo de vida dos contêineres
5. **Limpe volumes não utilizados** com `docker volume prune`
