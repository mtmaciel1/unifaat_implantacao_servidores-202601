## 1. Conceitos Fundamentais

### O Problema: Gerenciar Múltiplos Containers é Como Fazer Malabarismo

Imagine que você precisa preparar um jantar completo: entrada, prato principal, sobremesa e bebidas. Cada prato tem seu tempo de preparo, alguns dependem de outros, e tudo precisa ficar pronto ao mesmo tempo. Fazer isso sozinho, controlando cada panela individualmente, é estressante e propenso a erros.

É exatamente isso que acontece quando você tem uma aplicação moderna com múltiplos containers:
- **Frontend React** (precisa da API)
- **Backend API** (precisa do banco de dados)
- **Banco PostgreSQL** (precisa estar pronto primeiro)
- **Cache Redis** (usado pela API)
- **Proxy Nginx** (precisa que tudo esteja funcionando)

Gerenciar cada container individualmente com comandos `docker run` é como tentar cozinhar esse jantar controlando cada panela separadamente - você vai se perder!

### Docker Compose: O Chef Profissional

**Docker Compose** é como contratar um chef profissional que conhece a receita completa. Você escreve o "cardápio" uma vez (arquivo YAML) e ele coordena todo o preparo automaticamente.

**Analogia Completa:**
- **Receita (docker-compose.yml)**: O plano de como preparar tudo
- **Chef (Docker Compose)**: Quem executa a receita
- **Pratos (Services)**: Cada container da aplicação
- **Cozinha (Network)**: Onde tudo se comunica
- **Despensa (Volumes)**: Onde os ingredientes ficam guardados

### Por que Docker Compose Existe?

#### Problemas que Resolve:

**1. Complexidade de Comandos**
Sem Compose, você precisa lembrar de dezenas de parâmetros para cada container. Com Compose, tudo fica documentado no arquivo YAML.

**2. Ordem de Inicialização**
Alguns containers dependem de outros. O banco precisa estar pronto antes da API. Docker Compose gerencia essas dependências automaticamente.

**3. Comunicação entre Containers**
Containers precisam se "encontrar" na rede. Docker Compose cria automaticamente uma rede onde todos se comunicam pelo nome.

**4. Reprodutibilidade**
Sua aplicação funciona igual em qualquer máquina. Basta ter o arquivo `docker-compose.yml` e executar `docker-compose up`.

**5. Versionamento**
O arquivo YAML pode ser commitado no Git junto com o código, mantendo a infraestrutura versionada.

### Conceitos Fundamentais do Docker Compose

#### 1. Declarativo vs Imperativo

**Imperativo (comandos docker run)**: Você diz **COMO** fazer
```bash
# Você precisa saber a ordem, os parâmetros, as dependências...
docker network create...
docker run postgres...
docker run redis...
docker run api...
```

**Declarativo (Docker Compose)**: Você diz **O QUE** quer
```yaml
# Você descreve o resultado final desejado
services:
  postgres: ...
  redis: ...
  api: ...
```

#### 2. Services (Serviços)

Um **Service** no Docker Compose é como um "papel" na sua aplicação:
- **web**: O servidor web (nginx, apache)
- **api**: A API backend (Node.js, Python, Java)
- **database**: O banco de dados (PostgreSQL, MySQL)
- **cache**: O sistema de cache (Redis, Memcached)

Cada service se torna um container, mas com superpoderes:
- **Nome fixo**: Outros services sempre encontram pelo nome
- **Configuração centralizada**: Tudo definido no YAML
- **Dependências automáticas**: Docker Compose sabe a ordem de iniciar

#### 3. Networks (Redes)

Por padrão, Docker Compose cria uma **rede privada** onde todos os services se comunicam:
- Services se "enxergam" pelo nome
- Isolamento do mundo exterior
- Comunicação segura e automática

É como uma **rede interna de uma empresa** - todos os departamentos se comunicam, mas o mundo exterior não tem acesso direto.

#### 4. Volumes (Persistência)

Volumes garantem que dados importantes não se percam:
- **Dados do banco**: Sobrevivem mesmo se o container for removido
- **Arquivos de configuração**: Podem ser editados no host
- **Logs da aplicação**: Ficam acessíveis para análise

### Anatomia do docker-compose.yml

#### Estrutura Básica
```yaml
version: '3.8'          # Versão do formato

services:               # Definição dos containers
  web:                  # Nome do serviço
    image: nginx        # Imagem a usar
    ports:              # Portas a expor
      - "80:80"
    
  database:             # Outro serviço
    image: postgres:13
    environment:        # Variáveis de ambiente
      POSTGRES_PASSWORD: senha

volumes:                # Volumes para persistência
  db-data:

networks:               # Redes personalizadas (opcional)
  frontend:
```

#### Principais Configurações de um Service

**Image vs Build**
- `image`: Usa imagem pronta (nginx, postgres, redis)
- `build`: Constrói imagem a partir de Dockerfile local

**Ports (Mapeamento de Portas)**
- Conecta portas do container com o host
- Formato: `"porta_host:porta_container"`

**Environment (Variáveis de Ambiente)**
- Configurações que o container precisa
- Senhas, URLs de conexão, configurações

**Volumes (Persistência de Dados)**
- Conecta pastas do host com o container
- Garante que dados não se percam

**Depends_on (Dependências)**
- Define ordem de inicialização
- Garante que serviços necessários subam primeiro

### Comandos Essenciais do Docker Compose

#### Os "Big 3" - Comandos que Você Mais Vai Usar

**1. `docker-compose up`** - "Ligar tudo"
- Lê o arquivo `docker-compose.yml`
- Cria redes, volumes e containers necessários
- Inicia tudo na ordem correta
- `-d`: Roda em background (detached)

**2. `docker-compose down`** - "Desligar tudo"
- Para todos os containers
- Remove containers e redes
- Mantém volumes (dados seguros)
- `-v`: Remove volumes também (CUIDADO!)

**3. `docker-compose logs`** - "O que está acontecendo?"
- Mostra logs de todos os services
- `-f`: Acompanha logs em tempo real
- `logs web`: Logs apenas do service "web"

#### Comandos de Desenvolvimento

**4. `docker-compose ps`** - "Status da aplicação"
- Mostra quais containers estão rodando
- Estado de cada service (Up, Down, Restarting)

**5. `docker-compose exec`** - "Entrar no container"
- Executa comandos dentro de um container rodando
- `exec web bash`: Abre terminal no container "web"

**6. `docker-compose build`** - "Reconstruir imagens"
- Reconstrói imagens que usam `build`
- Útil quando você muda o Dockerfile

#### Comandos de Manutenção

**7. `docker-compose restart`** - "Reiniciar services"
- Reinicia containers sem recriar
- Útil para aplicar mudanças de configuração

**8. `docker-compose pull`** - "Atualizar imagens"
- Baixa versões mais recentes das imagens
- Mantém aplicação atualizada

### Exemplo Prático: Blog Simples

Vamos criar um blog com WordPress e MySQL:

```yaml
version: '3.8'

services:
  # Banco de dados
  database:
    image: mysql:8.0
    environment:
      MYSQL_DATABASE: wordpress
      MYSQL_USER: wp_user
      MYSQL_PASSWORD: wp_pass
      MYSQL_ROOT_PASSWORD: root_pass
    volumes:
      - db_data:/var/lib/mysql

  # WordPress
  wordpress:
    image: wordpress:latest
    ports:
      - "8080:80"
    environment:
      WORDPRESS_DB_HOST: database:3306
      WORDPRESS_DB_NAME: wordpress
      WORDPRESS_DB_USER: wp_user
      WORDPRESS_DB_PASSWORD: wp_pass
    depends_on:
      - database
    volumes:
      - wp_data:/var/www/html

volumes:
  db_data:
  wp_data:
```

**Para usar:**
```bash
# Subir o blog
docker-compose up -d

# Acessar: http://localhost:8080

# Ver logs
docker-compose logs -f

# Parar tudo
docker-compose down
```

### Ambientes Diferentes: Desenvolvimento vs Produção

#### Conceito de Override Files

Docker Compose permite ter **configurações base** + **configurações específicas** para cada ambiente:

**docker-compose.yml** (base - comum a todos ambientes)
```yaml
version: '3.8'
services:
  app:
    build: .
    environment:
      NODE_ENV: production
```

**docker-compose.override.yml** (desenvolvimento - aplicado automaticamente)
```yaml
version: '3.8'
services:
  app:
    volumes:
      - .:/app          # Código em tempo real
    environment:
      NODE_ENV: development
      DEBUG: true
```

**docker-compose.prod.yml** (produção - aplicado manualmente)
```yaml
version: '3.8'
services:
  app:
    restart: unless-stopped
    environment:
      NODE_ENV: production
```

**Como usar:**
```bash
# Desenvolvimento (usa override automaticamente)
docker-compose up

# Produção (especifica arquivo)
docker-compose -f docker-compose.yml -f docker-compose.prod.yml up
```

### Boas Práticas

#### 1. Use Arquivo .env para Secrets
Em vez de colocar senhas no YAML, use arquivo `.env`:

**.env**
```
DB_PASSWORD=senha_super_secreta
API_KEY=chave_da_api
```

**docker-compose.yml**
```yaml
services:
  database:
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}
```

#### 2. Defina Health Checks
Garanta que services estejam realmente prontos:

```yaml
services:
  database:
    image: postgres:13
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 30s
      timeout: 10s
      retries: 3
```

#### 3. Use Restart Policies
Para produção, configure reinicialização automática:

```yaml
services:
  app:
    restart: unless-stopped  # Reinicia se falhar
```

### Troubleshooting Comum

#### "Port already in use"
**Problema**: Porta já está sendo usada por outro processo
**Solução**: Mude a porta no docker-compose.yml ou pare o processo conflitante

#### "Service 'web' failed to build"
**Problema**: Erro na construção da imagem
**Solução**: Verifique o Dockerfile e use `docker-compose build --no-cache`

#### "Cannot connect to database"
**Problema**: Aplicação não consegue conectar ao banco
**Solução**: Verifique se usou o nome correto do service e se o banco está pronto

### Docker Compose vs Kubernetes

#### Quando Usar Docker Compose
- **Desenvolvimento local**: Perfeito para ambiente de dev
- **Aplicações simples**: Poucos containers, um servidor
- **Prototipagem**: Testar ideias rapidamente
- **CI/CD**: Testes automatizados

#### Quando NÃO Usar Docker Compose
- **Produção em escala**: Muitos servidores, alta disponibilidade
- **Orquestração complexa**: Auto-scaling, rolling updates
- **Multi-cloud**: Aplicação distribuída em várias clouds

**Resumo**: Docker Compose é excelente para desenvolvimento e aplicações simples. Para produção complexa, considere Kubernetes ou Docker Swarm.

## 2. Questões de Múltipla Escolha

### Questão 1
Qual é a principal vantagem do Docker Compose em relação ao gerenciamento manual de containers com comandos `docker run`?

a) Docker Compose é mais rápido na execução dos containers
b) Docker Compose permite definir toda a infraestrutura em um arquivo declarativo e reproduzível
c) Docker Compose consome menos recursos do sistema
d) Docker Compose só funciona em ambientes de produção

**Gabarito: B**

**Justificativa:** A principal vantagem do Docker Compose é permitir definir toda a infraestrutura de uma aplicação multi-container em um arquivo YAML **declarativo** (você descreve o que quer, não como fazer) e **reproduzível** (funciona igual em qualquer máquina). Isso elimina a complexidade de múltiplos comandos `docker run`, torna o ambiente documentado e versionável, e facilita o compartilhamento entre desenvolvedores.

### Questão 2
No Docker Compose, qual é a diferença entre usar `image` e `build` na definição de um serviço?

a) `image` é para desenvolvimento e `build` é para produção
b) `image` especifica uma imagem pronta de um registry, `build` constrói a imagem a partir de um Dockerfile
c) `image` é mais seguro que `build`
d) Não há diferença prática entre eles

**Gabarito: B**

**Justificativa:** `image` especifica uma imagem pronta que será baixada de um registry (Docker Hub, ECR, etc.), como `nginx:alpine` ou `postgres:13`. `build` instrui o Compose a construir uma imagem localmente a partir de um Dockerfile no contexto especificado. Use `image` para serviços baseados em imagens prontas e `build` para sua aplicação personalizada que precisa ser compilada.

### Questão 3
Você quer garantir que o serviço "web" só inicie depois que o serviço "database" estiver rodando. Qual configuração deve usar?

a) `links: [database]`
b) `depends_on: [database]`
c) `networks: [database]`
d) `volumes_from: [database]`

**Gabarito: B**

**Justificativa:** `depends_on` define a ordem de inicialização dos serviços, garantindo que o serviço "database" seja iniciado antes do "web". Importante notar que isso apenas controla a ordem de start dos containers, não garante que o database esteja pronto para receber conexões - para isso seria necessário usar healthchecks. `links` é legado (deprecated), `networks` define redes de comunicação, e `volumes_from` compartilha volumes.
